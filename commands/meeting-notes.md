读取会议录音/妙记链接，自动识别会议类型，输出结构化摘要+人员 Todo，或用户访谈纲要+研发速查。

输入：$ARGUMENTS（飞书妙记 URL、腾讯会议链接，或逐字稿文本）

> **前置依赖**
> - 飞书妙记：需安装 `lark-cli` 并完成飞书授权
> - GitHub 上传：需安装 `gh` 并完成 GitHub 授权（`gh auth login`）
> - 腾讯会议链接：需安装 gstack browse 工具，或直接粘贴逐字稿文本

---

## 两种模式

| 模式 | 触发条件 | 输出 |
|------|---------|------|
| A：团队会议 | 参会人全是团队成员，议题是产品/运营/复盘等 | 主题分区摘要 + 人员 Todo 分配 + 待决策事项 |
| B：用户访谈 | 标题/内容含"用户访谈"，或有明显外部参与者 | 访谈纲要（可选上传 GitHub）+ 研发速查 |

无法判断时，询问用户："这是团队会议还是用户访谈？"

## 第一步：获取会议内容

**路径 A：飞书妙记 URL（含 `/minutes/`）**

检查 memory 中是否已有用户姓名 + wake word 配置。如果没有，先获取：
```bash
lark-cli api GET /open-apis/authen/v1/user_info --as user
```
询问 wake word 设置（默认为空，不启用），记录到 memory 后不再重复询问。

```bash
TOKEN=$(echo "$URL" | sed 's/.*\/minutes\///' | cut -d'?' -f1)
lark-cli minutes minutes get --params "{\"minute_token\": \"$TOKEN\"}"
lark-cli vc +notes --minute-tokens $TOKEN
```

读取返回的 transcript.txt 完整内容。

**路径 B：飞书文档 URL（含 `/docx/` 或 `/wiki/`）**

```bash
lark-cli docs +fetch $URL
```

在文档末尾找逐字稿链接，如果是 `/docx/` 链接再 fetch 一次，如果是 `/minutes/` 链接走路径 A。如果都没有，直接基于文档内容分析。

**路径 C：腾讯会议 URL**

用 browse 工具读取（页面需要 JS 渲染）：

```bash
B="$HOME/.claude/skills/gstack/browse/dist/browse"
$B goto "$URL"
sleep 2
$B wait --networkidle
$B text 2>&1 | head -600
$B scroll --direction down --amount 3000
sleep 1
$B text 2>&1 | tail -500
$B scroll --direction down --amount 3000
sleep 1
$B text 2>&1 | tail -500
```

如果 browse 失败或需要登录，告知用户：
> "页面需要登录才能访问。请把逐字稿文本直接粘贴过来，我来整理。"

**路径 D：用户直接粘贴逐字稿**

直接使用，跳到 PREFLIGHT。

## PREFLIGHT：获取内容后必做，整理前先扫描

获取到逐字稿后，在进入任何整理步骤前，先做以下四项扫描。扫描结果在内部使用，不需要输出给用户，直接应用后继续。

### P1：发言人识别

列出逐字稿中所有发言人标签（"Speaker 1"、"SY"、拼音缩写等），尝试从上下文推断真实姓名和角色。

- 能确认的：直接映射，后续整理统一用真实姓名
- 无法确认的：标注为"[未知发言人]"，如果涉及关键行动项，在输出时标注"（发言人待确认）"
- 被提及但未参会的人（如"让小王做一下"）：单独记录，后续加入行动项时标注负责人

### P2：拼写变体统一

AI 转录经常把同一个名字/产品写出多种形式。扫描全文，找出所有变体，统一为一个标准写法后继续：

常见变体类型：
- 人名（同一个人被写成英文名、中文名、拼音缩写等不同形式）
- 产品名（拼写错误、中英混用）
- 公司/品牌缩写（中英文混用同一概念）

统一后全文使用标准写法，不再使用变体。

### P3：偏题内容识别

以下内容静默排除，不进入整理结果：
- 开场寒暄、闲聊、道别
- 掉线、重连、技术问题
- 背景噪声、无意义插话

### P4：未解决议题标记

扫描全文，找出所有"讨论了但没拍板"的事项，标记为 Outstanding（不同于已分配的 Todo）。判断标准：
- 有讨论但没有明确的决策人或结论
- 说了"待定"、"再看看"、"下次讨论"、"需要确认"
- 涉及多方意见分歧、没有结论

PREFLIGHT 完成后，进入对应模式的整理步骤。

## 第二步（模式 A）：团队会议整理

### 2A-1：主题分区摘要

按自然议题分块，每块格式：

```
## [议题名]

- [核心决策或结论]
- [具体行动项，越具体越好]
- [待定/需跟进的事项（标明谁在跟进）]
```

议题名跟随会议实际内容，不要强行合并不同话题。

### 2A-2：人员 Todo 分配

扫描全文，按人名归集所有分配给他/她的事项：

```
## 人员 Todo

### @[姓名]

本周完成：
- [ ] [具体任务]（如有截止时间或验收标准，一并写出）
- [ ] ...

待确认：
- [需要这个人回复确认的事]

### @[姓名]

...

### 全团队

- [ ] [所有人都需要做的事]
```

**铁律：**
- Todo 要具体到可以直接行动，不能只写方向性描述
- 截止时间如果会上提到了，必须写
- 分歧未定的写"待确认"，不要硬造结论

### 2A-3：Outstanding Items（未决议题）

将 PREFLIGHT P4 中标记的未解决议题单独列出，不混入 Todo：

```
## 待决策事项

- [ ] [议题]（背景：[简述讨论内容]，卡点：[为什么没拍板]）
- [ ] ...
```

这些事项不分配给具体人，下次会议前需要有人推进决策。

### 2A-4：Wake Word 行动项扫描

扫描用户本人的所有发言，找出包含 wake word 的句子（如果设置了 wake word）。这些是用户在会上直接给 Agent 的指令，置顶展示：

```
## 用户直接指令（Wake Word 触发）

1. [指令内容] — 方案：[Agent 打算怎么执行]
```

### 2A-5：展示并确认执行

输出以上内容后，列出所有**可执行**的行动项编号，问用户：
> "需要我帮你执行哪些？直接说编号就行，比如"1 3 你来做"或"全部你来做"。"

执行工具：
- **约会议**：`lark-cli calendar +suggestion → +create`（飞书用户）
- **写文档**：`lark-cli docs +create`（飞书用户）
- **查文档/信息**：`WebSearch` 或 `WebFetch`
- **发消息**：起草内容，用 pbcopy 复制到剪贴板，告知在哪个群粘贴（不代替用户发送）

## 第二步（模式 B）：用户访谈整理

### 2B-0：首次配置（仅第一次运行时）

检查 memory 中是否已有以下配置，没有则逐一询问并记录（后续不再重复询问）：

1. **产品名**：你的产品叫什么？（用于文件名和模板标题）
2. **GitHub 上传**：访谈纲要要上传到 GitHub 吗？
   - 如果是：GitHub repo 路径是什么？（格式：`owner/repo`）目标目录？（如 `docs/interviews`）
   - 如果否：仅在对话中展示，不上传

配置存入 memory，key 为 `meeting_notes_config`。

### 2B-1：确认受访者标识

如果逐字稿无法识别，问一次：
> "受访者叫什么（用于文件名）？"

其他信息从逐字稿里提取，不主动问。

### 2B-2：确定文件编号（仅当配置了 GitHub 上传时）

```bash
REPO="[memory中的repo]"
DIR="[memory中的目录]"
gh api repos/$REPO/contents/$DIR 2>/dev/null \
  | python3 -c "
import json, sys, re
try:
    items = json.load(sys.stdin)
    if isinstance(items, list):
        nums = [int(m.group(1)) for f in items if (m := re.match(r'^(\d+)-', f['name']))]
        print(max(nums) + 1 if nums else 1)
    else:
        print(1)
except:
    print(1)
"
```

如果未配置 GitHub 上传，编号从 1 开始自增（仅用于文件名）。

### 2B-3：生成访谈纲要

逐字稿原话 > AI 纪要。每个重要结论必须有原话引用（`> 「...」`）。

```markdown
# NN-[产品名]用户访谈-[受访者标识]

**受访者**：[标识] ｜ **身份**：[职业/行业] ｜ **参与者**：[团队成员] ｜ **时长**：约 XX 分钟 ｜ **日期**：YYYY-MM-DD

---

## 受访者背景

[3-4 段自然段。职业背景、团队规模、核心工作场景（越具体越好）。使用产品的时长和活跃程度。来使用这个产品的核心动机。]

---

## 核心使用场景

### 场景一：[名称]

**使用方式**：[怎么用的？哪些功能？]
**触发时机**：[什么情况下用？]
**具体产出**：[做出来了什么？效果如何？]
**与其他工具对比**：[用户怎么评价 vs 其他工具，尽量用原话]

> 「[用户关键原话]」

**卡点**：[这个场景遇到的具体问题]

[根据访谈实际场景数量增减 ### 场景]

---

## Aha Moment 与卡点

**最爽的瞬间**：[具体场景+原话]

**最大卡点**：[让用户最不爽的体验+原话]

> 「[原话]」

---

## 产品认知

[用户怎么定位这个产品？与其他工具是替代还是互补？引用原话]

> 「[关键原话]」

---

## 问题反馈与需求

| 类别 | 详情 |
|------|------|
| Bug（高优先级） | 1. [现象+影响范围]<br>2. ...<br>3. ... |
| 认知卡点 | 1. [用户误解了什么+在哪里卡住]<br>2. ...<br>3. ... |
| 功能需求 | 1. [需求+场景]<br>2. ...<br>3. ... |

---

## 付费意愿

- **当前付费现状**：[为哪些工具付费+金额]
- **[产品名] 付费价位**：[具体数字+货币单位]
- **付费前提**：[什么条件才愿意付费]

> 「[付费相关原话]」
```

**整理铁律：**
1. 每个重要结论必须有逐字稿原话支撑，用 `> 「...」` 引用
2. 场景要写清楚使用方式、触发时机、产出、竞品对比
3. 未使用功能写真实原因，不能只写"未探索"，要找前置条件
4. 竞品点名道姓，不模糊化
5. 付费意愿要有数字，缺失的标注"（访谈未覆盖）"
6. 不编造内容

### 2B-4：生成研发速查

这是专门给研发看的 2 分钟速览，痛点直接映射到可执行改动。整体控制在 30-40 行以内。

```markdown
## 研发速查 | [受访者标识] 访谈

**一句话总结**：[用户身份]，[核心诉求]，[留存/流失]的核心原因是[原因]。

---

### 高优 Bug（本期必修）

| # | 问题 | 复现路径 | 负责模块 | 严重程度 |
|---|------|---------|---------|---------|
| B1 | [具体现象] | [操作步骤] | [前端/后端/具体功能] | 高/中/低 |
| B2 | ... | ... | ... | ... |

严重程度判断标准：高 = 核心功能不可用或直接导致流失；中 = 影响体验但有 workaround；低 = 轻微不便。

---

### 认知偏差（需要 Onboarding/Copy 改造）

| 功能 | 用户的错误认知 | 建议改法 | 提及次数 |
|------|-------------|---------|---------|
| [功能名] | 「[用户原话]」 | [具体改动：改 copy/加引导卡/调整 UI] | [N 次] |

---

### 功能需求（频率 × 优先级）

| 优先级 | 需求 | 用户场景 | 参考方案 | 提及次数 |
|--------|------|---------|---------|---------|
| P0 | [需求名] | [具体场景] | [简短建议] | [N 次] |
| P1 | ... | ... | ... | ... |
| P2 | ... | ... | ... | ... |

优先级判断：P0 = 直接影响付费转化或留存；P1 = 高频痛点但有 workaround；P2 = 锦上添花。

---

### 留存风险

[用户当前状态：活跃/冷淡/即将流失。核心原因一句话。如果有，说明一个最高优先级行动能挽回用户。]
```

**研发速查铁律：**
- Bug 表格必须有复现路径，不能只写现象
- 认知偏差必须给出具体改法，不能只说"需要改进"
- 功能需求必须说场景，不能只写功能名称
- 整体控制在 1 屏以内，研发不需要读纲要就能 get 核心

### 2B-5：上传访谈纲要到 GitHub（仅当配置了上传时）

```bash
REPO="[memory中的repo]"
DIR="[memory中的目录]"
FILENAME="NN-[产品名]用户访谈-[受访者标识].md"
ENC=$(python3 -c "import urllib.parse; print(urllib.parse.quote('$FILENAME'))")

# 写入临时文件
cat > /tmp/interview_draft.md << 'HEREDOC'
[访谈纲要内容，不含研发速查]
HEREDOC

CONTENT=$(base64 < /tmp/interview_draft.md)

EXISTING=$(gh api repos/$REPO/contents/$DIR/$ENC 2>/dev/null \
  | python3 -c "import json,sys; d=json.load(sys.stdin); print(d.get('sha',''))" 2>/dev/null || echo "")

if [ -n "$EXISTING" ]; then
  gh api repos/$REPO/contents/$DIR/$ENC \
    --method PUT --field message="feat: update $FILENAME" \
    --field "content=$CONTENT" --field encoding="base64" --field sha="$EXISTING"
else
  gh api repos/$REPO/contents/$DIR/$ENC \
    --method PUT --field message="feat: add $FILENAME" \
    --field "content=$CONTENT" --field encoding="base64"
fi
```

如果未配置上传，跳过此步骤。

### 2B-6：输出结果

完成后输出：
1. GitHub 文件直链（如果上传了）
2. **完整研发速查**（直接展示在对话中，方便复制进研发群）
3. 3 条核心 insight（聚焦场景和产品决策）

然后问用户：
> "研发速查要发给研发群吗？我可以帮你起草消息。"

## 错误处理

| 错误 | 处理 |
|------|------|
| 飞书未授权 | 提示运行 `lark-cli auth login` |
| 页面需要登录 | 提示用户粘贴逐字稿文本 |
| 转录内容不完整 | 整理已有内容，空白部分标注"（访谈未覆盖）" |
| GitHub 未授权 | 提示运行 `gh auth login`，或跳过上传直接输出 Markdown |
| GitHub API 失败 | 把 Markdown 输出在对话框，让用户手动上传 |
| 文件名冲突 | 自动递增编号 |
| 无法判断会议类型 | 询问用户 |

---

## 注意事项

- 逐字稿发言人名字可能是拼音或英文名，尽量匹配到真实姓名
- 时间戳格式是"分:秒"，引用时写成"XX分XX秒"方便阅读
- 行动项穷尽，宁可多列不要遗漏
- 不编造内容，逐字稿没有的标注清楚
- 不要用破折号（——），用逗号或句号代替
- 输出语言跟随用户对话语言
